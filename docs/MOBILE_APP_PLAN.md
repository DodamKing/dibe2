Flutter 채택(Capacitor 대비 `audio_service`+`just_audio` 성숙도 우위), **Android 전용**(Windows 개발 환경이라 iOS 빌드 불가), A안 우선 진행으로 확정. 별도 저장소 `d:\projects\dibe2-app`에 프로젝트 생성 완료 — 자세한 진행 상황은 그 저장소의 `docs/PLAN.md`, `docs/WORK_LOG.md` 참고. 상세 내용은 `docs/WORK_LOG.md`의 2026-07-07 항목 참고

## 배경
- 현재 웹앱에서는 모바일 백그라운드 재생이 불가능 (브라우저 제약)
- 모바일에서 음악을 듣기에는 앱이 더 적합
- 핵심 목표: **백그라운드 재생 + 잠금화면 컨트롤**

## 기술 판단

### WebView만으로는 백그라운드 재생 불가
- WebView 안의 YouTube IFrame API는 앱이 백그라운드로 가면 OS가 suspend시킴
- iOS/Android 모두 동일한 제약

### 해결 방법
- Flutter `just_audio` + `audio_service` 패키지로 **네이티브 오디오 재생**
- 백그라운드 재생, 잠금화면 컨트롤, 알림바 컨트롤 모두 지원됨

> 🔴 **오디오 스트림을 어디서 받을지가 미해결이다** (2026-07-17 확인)
>
> 이 문서는 원래 "Invidious 프록시에서 오디오 스트림 URL 획득 (기존 구현 활용)"이라고 적고 있었으나 **둘 다 사실이 아니다**:
> - **기존 구현이 없다.** `advancedInvidiousManager.js` / `/stream/:songId` 는 이미 삭제됐다
> - **Invidious 우회는 실패로 판명났다** (`WORK_LOG` 2026-05-24): anti-bot 페이지 + `X-Frame-Options: SAMEORIGIN`, 그리고 **TJ/한국 라이선스 영상은 유럽·북미 IP 인스턴스가 stream 추출 자체를 못 한다**
> - 대안으로 검토했던 **자체 서버 yt-dlp 추출도 "trade-off 안 맞음"으로 접었다**(한국 IP 호스팅 + 별도 서버 + 봇 차단 위험)
>
> 즉 **A안의 "재생만 네이티브로"라는 전제가 아직 안 풀렸다.** 모바일 앱을 재개하기 전에 이 결정부터 해야 한다.
> (참고: 웹 비디오 탭은 이 문제를 "백그라운드 재생을 포기하고 Media Session API만 연동"으로 우회했다 — `WORK_LOG` 2026-06-28)

## 두 가지 접근법

### A안: WebView 하이브리드 (추천 — 우선 진행)
- 기존 웹앱을 WebView로 띄우고, **재생만 네이티브로 처리**
- JS Bridge로 WebView ↔ Flutter 통신
- 장점:
  - 기존 코드 최대한 재활용
  - 개발 속도 빠름
  - **웹 코드 수정 시 앱 업데이트 불필요** (네이티브 재생 로직 변경 시에만 필요)
- 단점:
  - WebView 특유의 어색함 (느린 전환, 네이티브 느낌 부족)

### B안: Flutter 네이티브 UI
- UI를 Flutter로 새로 만들고, 백엔드 API만 그대로 활용
- 장점:
  - 네이티브 UX (빠른 전환, 제스처, 애니메이션)
  - 오프라인 캐싱 등 네이티브 기능 확장 용이
- 단점:
  - UI 전체를 새로 구현해야 함 (공수 큼)
  - 모든 UI 변경이 앱 업데이트 필요

### 비교표

| | A. WebView 하이브리드 | B. Flutter 네이티브 |
|---|---|---|
| 개발 속도 | 빠름 (재생만 구현) | 느림 (UI 전체 구현) |
| 앱 업데이트 | 거의 불필요 | UI 변경마다 필요 |
| UX 품질 | 웹 느낌 | 네이티브 느낌 |
| 백그라운드 재생 | 가능 | 가능 |
| 잠금화면 컨트롤 | 가능 | 가능 |

### 전략
A안으로 빠르게 만들고, 불편한 부분을 점진적으로 B안(네이티브 UI)으로 교체

## 프로젝트 구조

**별도 프로젝트로 분리**

```
dibe2/          ← 기존 (웹앱 + API, Netlify 배포)
dibe2-app/      ← 새 프로젝트 (Flutter 앱)
```

- Flutter 프로젝트는 자체 구조(`lib/`, `android/`, `ios/`, `pubspec.yaml`)를 가짐
- 웹앱 프로젝트와 섞으면 빌드/배포가 복잡해짐
- 앱에서는 기존 Netlify API를 HTTP로 호출 (백엔드 코드 중복 없음)

## 고려사항
- **CORS**: Netlify API가 앱에서의 요청도 허용하도록 설정 필요할 수 있음
- ~~**Invidious 스트림 URL 수명**: URL 만료 처리 필요~~ → 위 "해결 방법"의 경고 참조. Invidious 경로 자체가 막혔다
- **YouTube ToS**: 공식 앱 외 백그라운드 재생은 정책 위반 소지
